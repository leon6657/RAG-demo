# RAG（检索增强生成）原理

## 什么是 RAG

RAG（Retrieval-Augmented Generation）是一种将信息检索与大语言模型生成能力相结合的架构。它的核心思想是：在 LLM 生成回答之前，先从外部知识库中检索相关文档片段，将这些片段作为上下文提供给 LLM，从而让回答基于事实而非模型记忆。

## RAG 全链路

RAG 系统的工作流程分为三个主要阶段：

### 1. 索引阶段（Indexing）

- **文档加载**：从不同来源（Markdown、PDF、网页等）读取原始文档
- **文档分块**：将长文档切分为更小的片段，常用策略包括按标题层级切分、按段落切分、按固定 token 数量切分
- **向量化 Embedding**：使用 Embedding 模型将每个文本块转换为向量（高维浮点数数组）
- **向量存储**：将向量存入向量数据库（如 ChromaDB、Qdrant、Pinecone），构建索引以便快速检索

### 2. 检索阶段（Retrieval）

- 将用户查询同样通过 Embedding 模型转为向量
- 在向量数据库中执行相似度搜索（常用余弦相似度或欧几里得距离）
- 返回 Top-K 个最相似的文档片段及其相似度分数

### 3. 生成阶段（Generation）

- 将检索到的文档片段作为上下文，与用户问题拼接成 Prompt
- 送入 LLM 生成最终回答
- 好的 Prompt 设计会要求 LLM 严格基于上下文回答，上下文不足时明确表示不知道

## 为什么需要 RAG

LLM 存在两大固有缺陷：

- **幻觉（Hallucination）**：模型可能编造看起来合理但实际错误的信息
- **知识过时**：模型训练数据有截止日期，无法获取最新信息

RAG 通过引入外部知识源，同时缓解了这两个问题，并且不需要重新训练模型。

## 朴素 RAG vs 高级 RAG

朴素 RAG 即上述的标准流程，而高级 RAG 在此基础上增加了查询改写、重排序、混合检索、Hypothetical Document Embeddings（HyDE）等优化手段，以进一步提升检索准确率和生成质量。
